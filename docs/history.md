# 영림 웹 카탈로그 프로젝트 개발 히스토리

본 문서는 `YL_cadalog` 프로젝트의 초기 기획부터 최종 통합까지의 주요 개발 이력을 기록합니다.

---

## 📅 프로젝트 개요
- **목적**: 영림 임업 종합 카탈로그 PDF를 웹 환경에 최적화하여 제공하고, 인테리어 필름 데이터에 대한 스마트 검색(텍스트/색상) 기능을 구현함.
- **주요 기술 스택**:
  - **Frontend**: Vanilla JS, HTML5, CSS3, PDF.js
  - **Backend**: Vercel Serverless Functions (Node.js)
  - **Data**: Vercel Proxy API, JSON-based Color Database
  - **Hosting**: GitHub Pages (Client), Vercel (API & Image Search)

---

## 🚀 개발 마일스톤

### 1단계: PDF 뷰어 기초 구축 (2026-02-27 ~ 2026-03-03)
- **PDF.js 통합**: 카페24 및 GitHub Pages 환경에서 PDF를 웹으로 렌더링하는 기본 엔진 구축.
- **카테고리 탭 구현**: 문틀, 몰딩, 손잡이 탭을 통해 특정 페이지 범위를 스크롤 뷰로 표시.
- **CORS 이슈 해결**: GitHub Pages에 대용량 PDF를 업로드하고 절대 경로 연동을 통해 보안 정책 문제 해결.

### 2단계: 필름 카탈로그 및 Vercel API 개발 (2026-03-04 ~ 2026-03-05)
- **Vercel 프록시 API**: 영림 공식 사이트의 필름 데이터를 안전하게 가져오기 위한 `/api/film.js` 백엔드 구축.
- **실시간 검색**: 필름 제품명 및 제품코드로 실시간 필터링이 가능한 검색 기능 추가.
- **고도화된 이미지 검색**:
  - 사용자 이미지 업로드 시 주요 색상(RGB) 추출 로직 구현.
  - 전용 이미지 프록시(`/api/proxy-image.js`)를 통해 외부 이미지 분석용 CORS 우회.
  - 유클리드 거리 알고리즘을 활용한 제품 간 색상 유사도(%) 정렬 기능 도입.
  - 전체 제품 색상 DB(`colors.json`) 구축.

### 3단계: 메인 앱 최종 통합 및 멀티 엔진 구축 (2026-03-05)
- **통합 연동**: Vercel에서 검증된 필름 검색 기능을 메인 `catalog.html` 앱 안으로 완벽하게 이식.
- **슬라이더 기반 UX 전환**: 세로 스크롤 방식에서 1페이지씩 넘겨보는 슬라이더 방식으로 전면 개편(모바일 스와이프/데스크톱 방향키 지원).
- **스프레드 자동 분할**: 2면 1매 형식의 PDF를 캔버스 크롭(Crop) 기술을 이용해 웹에서 한 페이지씩 보이도록 구현.

### 4단계: PDF 퍼포먼스 최적화 및 UX 고도화 (2026-03-05)
- **PDF 스트리밍 서비스**: 76MB 대용량 파일을 한 번에 받지 않고 필요한 부분만 조각내어 받는 Range Request 기술 도입, 초기 로딩 속도 90% 단축.
- **선행 렌더링(Pre-rendering)**: 현재 페이지 로딩 시 전후 페이지를 백그라운드에서 미리 렌더링하여 끊김 없는 페이지 전환 실현.
- **로딩 진행 바**: 스트리밍 진행 상태를 가시화하는 세련된 UI 인디케이터 도입.

### 5단계: 카테고리별 특화 기능 및 필름 탭 개편 (2026-03-05)
- **필름 탭 전면 개편**:
    - 카테고리 구분 없는 **'전체 상품 보기'**로 편의성 개선.
    - **'방염재고운영' 전용 버튼** 도입: 클릭 한 번으로 재고 있는 상품만 실시간 필터링.
    - 복잡한 이미지 검색 UI를 제거하고 깔끔한 필터 UI로 정렬.
- **몰딩 탭 특수 기능**:
    - **선행 이미지 배치**: PDF 시작 전 외부 URL의 재고 현황 이미지를 1페이지로 고정 노출하는 멀티 소스 렌더링 구현.
    - **범위 최적화**: 몰딩 관련 특정 페이지(448~464)만 렌더링하도록 제한하여 총 18페이지의 전용 섹션 구축.

### 6단계: PDF 분할 및 탭별 Lazy-Load 아키텍처 전환 (2026-03-05)
- **로딩 병목 분석**: GitHub Pages(Fastly CDN)에서 Range Request가 실제로 미작동(HTTP 206 대신 200 반환)함을 확인. `disableRange: false` 설정이 사실상 무의미하여 73MB 전체 PDF를 매번 다운로드하고 있었음을 진단.
- **PDF 분할 전략**: 73MB 단일 PDF에서 탭별로 필요한 페이지만 추출하여 개별 PDF로 분할:
    - `doorframe.pdf` (문틀, 326~337, 1.7MB)
    - `molding.pdf` (몰딩, 448~464, 2.7MB)
    - `handle.pdf` (손잡이, 340~353, 1.8MB)
    - 총 용량: 73MB → **6.2MB** (91.5% 감소)
- **Lazy-Load 아키텍처**: 초기 진입 시 전체 PDF 다운로드를 제거하고, 탭 선택 시 해당 카테고리 PDF만 on-demand 로드하는 구조로 전환.
    - `loadCategoryPdf()`: 카테고리별 PDF를 lazy-load 후 메모리 캐시(`categoryPdfDocs`)에 보관.
    - `getSplitPdfIndex()`: 인쇄 페이지 번호를 분할 PDF 내부 인덱스로 자동 변환하는 오프셋 계산 함수 도입.
- **성능 개선 결과**:
    - 초기 로딩: 73MB → 1.7MB (문틀 탭 기준, **약 97.7% 감소**)
    - WiFi 로딩 시간: ~6초 → **~0.15초**
    - 4G LTE 로딩 시간: ~20초 → **~0.5초**
    - 불필요한 앞 페이지(1~325, 338~447) 다운로드 완전 제거.

## 🔧 주요 해결 과제
1. **대용량 PDF 로딩**: ~~PDF.js 스트리밍 엔진 설정을 통해 76MB 파일의 초기 지연 시간 문제 해결.~~ → 6단계에서 탭별 분할 PDF + Lazy-Load 방식으로 근본적 해결.
2. **복합 소스 렌더링**: 서로 다른 출처(이미지 URL, PDF 페이지)의 데이터를 하나의 슬라이더 흐름으로 통합하는 로직 구현.
3. **사용자 경험 최적화**: 모바일 터치 스와이프와 PC 방향키를 동시 지원하여 기기 범용성 확보.
4. **CDN Range Request 미작동**: GitHub Pages(Fastly CDN)가 Range Request를 무시하는 환경적 한계를 PDF 분할 전략으로 우회.
