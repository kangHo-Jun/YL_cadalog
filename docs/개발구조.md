# 🏗️ 영림 웹 카탈로그 개발 구조 가이드

본 문서는 `YL_cadalog` 프로젝트의 기술적 구조를 설명하며, 새로운 기능을 추가하거나 유지보수하려는 개발자(또는 입문자)를 위한 지침서입니다.

---

## 1. 프로젝트 개요
> **"방대한 영림 카탈로그 PDF를 웹에서 빠르고 편리하게 조회하고, 필름 제품을 스마트하게 검색할 수 있는 웹 애플리케이션"**

---

## 2. 전체 파일 구조 및 역할

| 파일/폴더 | 역할 설명 |
| :--- | :--- |
| **`index.html`** * | 서비스의 메인 진입점. 탭 메뉴와 PDF가 그려질 틀을 제공합니다. |
| **`catalog.js`** | **핵심 두뇌.** PDF 로딩, 페이지 넘기기, 탭 전환, 검색 로직을 모두 담당합니다. |
| **`catalog.css`** | 디자인 담당. 슬라이드 애니메이션, 레이아웃, 모바일 대응 스타일을 포함합니다. |
| **`api/film.js`** | 필름 데이터를 가져오기 위한 다리 역할(Proxy API). Vercel에서 실행됩니다. |
| **`public/colors.json`** | 필름 색상 검색을 위한 RGB 데이터베이스입니다. |
| **`*.pdf`** | 각 카테고리별로 최적화(분할)된 실제 카탈로그 파일입니다. |

*\* 기존 `catalog.html`에서 이름이 변경되었습니다.*

---

## 3. 영림 카탈로그 → 내 카탈로그 변환 과정

### ① PDF에서 원하는 페이지만 추출하기
영림 공식 홈페이지의 전체 PDF는 용량이 너무 큽니다. 이를 해결하기 위해 필요한 페이지만 잘라냅니다.
1.  전체 PDF를 PDF 편집기(Acrobat, 웹 도구 등)로 엽니다.
2.  원하는 카테고리(예: 문틀)의 시작과 끝 페이지를 확인합니다.
3.  해당 범위만 별도 파일(예: `doorframe.pdf`)로 저장하여 저장소 루트에 업로드합니다.

### ② `CONFIG.mapping` 설정 방법
추출한 PDF의 실제 종이에 적힌 페이지 번호를 코드에 등록해야 합니다.
```javascript
// catalog.js 상단
mapping: {
    '문틀': [326, 327, 328], // 종이에 적힌 번호를 순서대로 적습니다.
    // 이 번호를 보고 프로그램이 PDF의 몇 번째 장인지 알아서 계산합니다.
}
```

## 4. 새로운 카테고리 추가 방법 ⭐
초보자도 따라 할 수 있는 3단계 과정입니다.

### 1단계: HTML에 버튼 추가
`index.html`의 `<ul id="tab-list">` 안에 새 메뉴를 만듭니다.
```html
<!-- index.html -->
<li class="tab-item" data-category="가구">가구</li>
```

### 2단계: PDF 정보 등록
`catalog.js`의 `CONFIG` 객체에 PDF 경로와 페이지 번호를 넣습니다.
```javascript
// catalog.js
const CONFIG = {
    categoryPdfs: {
        '가구': 'https://.../furniture.pdf', // 업로드한 PDF 주소
    },
    mapping: {
        '가구': [100, 101, 102], // 보여줄 페이지 번호들
    }
}
```

### 3단계: 로직 연결 (선택 사항)
만약 특별한 처리가 필요 없다면 자동으로 작동합니다.

## 5. 새로운 이미지/문서 삽입 방법 ⭐
PDF 파일이 아니더라도 공지사항이나 재고 현황 이미지를 탭 앞에 끼워 넣을 수 있습니다.

### 몰딩 탭 사례 (PDF 앞에 이미지 추가)
`catalog.js`의 `renderCategory` 함수 안에서 배열의 순서를 조정합니다.
```javascript
// catalog.js 내 renderCategory 함수
if (category === '가구') {
    currentCategoryPages = [
        // 이미지 타입을 가장 먼저 배치 (PDF보다 앞에 나옴)
        { type: 'IMAGE', url: 'https://.../notice.png' }, 
        // 그 뒤에 PDF 페이지들을 붙임
        ...pages.map(p => ({ type: 'PDF', num: p }))
    ];
}
```

## 6. 로딩 속도 개선 (퍼포먼스) 과정
사용자가 기다리지 않게 만드는 핵심 기술입니다.

*   **73MB → 6.2MB 분할**: 큰 파일 하나를 다 받는 대신, 탭별로 쪼개진 작은 파일만 다운로드합니다. (**Lazy-Load / 게으른 로딩**: 필요할 때만 불러옴)
*   **Range Request(범위 요청)**: PDF 전체을 다운로드하지 않고 현재 보려는 페이지만 조각내어 가져옵니다.
*   **선행 렌더링(Pre-rendering)**: 2페이지를 보고 있을 때, 3페이지를 보이지 않는 곳에서 미리 그려두어 넘길 때 끊김이 없게 합니다.

---

## 7. UI 및 동작 구조
*   **한 페이지씩 넘기기**: 2면 1매(양면) 형식의 PDF를 캔버스 기술로 반으로 잘라(**Crop**) 한 화면에 한 페이지만 나오게 합니다.
*   **탭 전환**: 상단 메뉴 클릭 시 기존 데이터를 지우고(`pageCache.clear()`) 새로운 카테고리의 PDF 정보를 즉시 불러옵니다.

## 8. GitHub Pages 배포 방법
수정한 내용을 인터넷에 적용하는 순서입니다.

1.  **파일 저장**: 수정한 모든 파일을 저장합니다.
2.  **커밋(Commit)**: "무엇을 수정했는지" 메모와 함께 기록을 남깁니다.
    - 예: `git commit -m "feat: 가구 카테고리 추가"`
3.  **푸시(Push)**: GitHub 서버로 파일을 보냅니다.
    - 예: `git push origin main`
4.  **확인**: 약 1~2분 뒤 본인의 GitHub Pages 주소로 접속하여 확인합니다.

---

> [!TIP]
> **전문 용어 사전**
> *   **CORS**: 보안상 이유로 다른 사이트의 데이터를 함부로 가져오지 못하게 하는 규칙 (프록시 API로 해결)
> *   **Canvas(캔버스)**: 웹에서 그림을 그리는 도구. PDF를 이미지로 변환하여 화면에 보여줄 때 사용합니다.
